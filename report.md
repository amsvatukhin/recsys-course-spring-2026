# Отчёт по ДЗ2

## 1. Abstract

Идея в том, чтобы не ломать уже работающий SasRec-I2I, а поверх него сделать второй этап: брать кандидатов из тех же в Redis и выбирать трек уже не "первый подходящий", а с помощью небольшой онлайн-модели. В контроле оставил штатный SasRec-I2I, в тритменте — свой класс в `online_mf.py`, разведение через уже существующий A/B в `server.py`. Модель после каждого шага подстраивается по тому, сколько времени пользователь прослушал предыдущую выдачу. Задумка была в том, чтобы вытянуть среднее время сессии относительно чистого SasRec-I2I.

## 2. Детали

Код лежит в `botify/botify/recommenders/online_mf.py`, подключение - в `server.py`: контроль это `I2IRecommender` по SasRec, тритмент - `OnlineMFRecommender`. Если истории нет или пул пустой, всё равно уходит в тот же SasRec, чтобы не было странных краевых случаев.

Сначала читается история из Redis. Первый кандидат собирается так же, как в SasRec: случайный anchor с весом по накопленному времени прослушивания, из списка i2i берётся первый ещё не слышанный трек. Дальше в пул добавляются другие треки из начала списков (не уезжаем далеко по рангу), чтобы было из чего выбрать, но не раздувать запрос. На каждого кандидата считается короткий числовой вектор признаков (ранг в списке, веса, время на предыдущем треке, совпадение anchor с предыдущим треком и несколько простых отображений id). Пока набралось мало наблюдений, ранжирование идёт по ручному prior; потом к нему подмешивается предсказание `SGDRegressor` из sklearn. Обучение - один `partial_fit` на пару (признаки последней выдачи, время прослушивания, обрезанное сверху).

## 3. Результаты A/B

Прогонял симулятор у себя в WSL2, seed как в Makefile (31312). Сохранял логи в разные `DATA_DIR`, потом `analyze_ab.py` писал `ab_result.json`. Ниже только то, что в задании считается успехом: `mean_time_per_session`, сравнение T1 с контролем.

| прогон | эпизоды | effect_pct | lower_pct | upper_pct | control_mean | treatment_mean | significant |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| data_dense8k | 8000 | 0.53 | -2.08 | 3.15 | 7.43 | 7.47 | false |
| data_base15k | 15000 | 0.83 | -1.38 | 3.03 | 7.27 | 7.33 | false |

По этим двум прогонам тритмент чуть выше по среднему, но доверительный интервал везде задевает ноль, то есть про статистически устойчивое улучшение говорить нельзя, формальное условие ДЗ не выполнено. Латентность запросов у тритмента при этом заметно хуже, это видно по той же выгрузке, если смотреть метрику `mean_request_latency`.

Повторить цифры можно так: `make setup`, затем `make EPISODES=8000 DATA_DIR=./data_dense8k run` и `make EPISODES=15000 DATA_DIR=./data_base15k run`.
